# Частина 1 - теоретичні питання

### 1. Чому аудіо-характеристики винесені в окремий об'єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?

Бо це логічно один блок: danceability, energy, loudness і компанія описують ту саму річ (як трек звучить) і використовуються разом. Тримати їх в audio_features зручно: одним полем у проєкції беру або ховаю всі фічі, верхній рівень документа не засмічений десятком метрик, і додати нову це просто додати ключ в один об'єкт.

Коли таке вкладення створює проблеми: якщо документ вкладено в 3+ рівні, запити стають нечитабельні; індекси по вкладених полях треба писати повним шляхом (audio_features.tempo). У нашій схемі рівень вкладеності всього один, тому проблем не виникає.

### 2. Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?

У вихідному CSV кілька артистів зберігаються в одному рядку через роздільник `;` (наприклад, `"Drake;Future"`). Тримати кілька значень в одному полі погано: пошук одного артиста перетворюється на $regex, який повільний і ненадійний. Масив натомість дає прямий пошук `{ artists: "Drake" }`, нормальний $unwind для топ-артистів (як у ч.2 завд.2 і ч.3 завд.1), multikey-індекс і $size для кількості співавторів.

### 3. Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?

$out повністю перезаписує цільову колекцію результатом пайплайну. $merge вливає документи інкрементально: є контроль через whenMatched і whenNotMatched, можна оновлювати, вставляти або ігнорувати.

У трансформації я використала $out, бо треба разово зібрати tracks з нуля з tracks_raw. $merge був би доречний, якби це був регулярний пайплайн, що дописує нові дані в існуючу таблицю (наприклад, щоденне оновлення materialized view), або якщо колекція використовується в продакшені і її не можна тимчасово втратити під час перебудови.

# Частина 2 - теоретичні питання

### 1. Для чого використовується інструкція $unwind?

Щоб розкласти масив на окремі документи, по одному на елемент. У завданні 2 artists це масив, а групувати треба по конкретному артисту. Без $unwind стадія $group згрупувала б за масивом цілком, тобто `["Drake", "Future"]` лежав би окремо від просто `["Drake"]`. Після $unwind кожен артист стає окремим документом і $group рахує статистику саме на рівні артиста.

### 2. Чим $stdDevPop відрізняється від $stdDevSamp?

Обидва це стандартне відхилення, різниця у знаменнику. $stdDevPop ділить на N (вся генеральна сукупність), $stdDevSamp на N-1 (вибірка, з якої оцінюємо параметр популяції, правка Бесселя). У завд.3 ми працюємо з усіма треками жанру з нашої бази, це не вибірка, тому $stdDevPop. На великих N різниця мізерна, на малих (<30) помітна.

# Частина 3 - теоретичні питання

### 1. У запиті 1 ми фільтруємо виконавців, у яких менше 5 треків. Як зміниться результат, якщо знизити поріг до 1? А що станеться, якщо вибирати виконавців із більш ніж 50 треками? Поясніть результат.

З порогом 1 топ перетвориться на список окремих хітів, а не виконавців: один трек з популярністю 95 і артист уже в топі з середньою 95. Але це не показник того, що сам виконавець популярний, це просто один випадковий вдалий трек.

З порогом 50+ навпаки, пройдуть тільки артисти з великими дискографіями. А у великих каталогах завжди є менш популярні треки: альбомні наповнювачі, експерименти, нішеві релізи. Вони тягнуть середню популярність вниз. У підсумку топ ранжується скоріше за продуктивністю, ніж за реальною популярністю: справжня зірка з 200 треками програє у середній менш відомому виконавцеві з 50 треками, де всі 50 сильні. Тобто поріг це баланс: занадто низький впускає випадкові хіти, занадто високий зміщує рейтинг у бік плідних виконавців, у яких середня занижена через велику кількість менш популярних треків у каталозі.

### 2. У запиті 3 ми фільтруємо жанри з менше ніж 100 треками. Чи зміниться результат, якщо знизити поріг до 50? Поясніть результат.

З 50 пройдуть додаткові малі жанри. На малій вибірці середнє нестабільне: жанр з 60 треками може випадково потрапити в топ за рахунок кількох сильних треків, які перетягують середнє. Поріг 100 це простий захист від такого шуму. У нашому датасеті (по ~1000 треків на жанр) перехід 50 на 100 на топ майже не вплине, зміняться хіба граничні позиції.

# Частина 4 - Індекси та оптимізація

> Усі запити виконуються в `queries/part4_indexes.js`. У README наведено інтерпретацію виводу `explain()`.

### Завдання 1. Складений індекс під фільтр + сортування

Запит:
```javascript
db.tracks.find({
  track_genre: "pop",
  "audio_features.danceability": { $gte: 0.7 }
}).sort({ popularity: -1 })
```

**До індексу** (фрагмент `explain("executionStats")`):

```
winningPlan.stage:           SORT
winningPlan.inputStage.stage: COLLSCAN
nReturned:                   354
totalDocsExamined:           113999      ← весь датасет
totalKeysExamined:           0
executionTimeMillis:         88
```

Mongo перебрав **усі 113 999 документів** і відсортував їх у пам’яті - індекс не задіяний (`COLLSCAN`).

**Створений індекс:**
```javascript
db.tracks.createIndex(
  { track_genre: 1, "audio_features.danceability": 1, popularity: -1 },
  { name: "idx_genre_dance_pop" }
);
```

Порядок полів обраний за правилом **ESR (Equality → Sort → Range)**: `track_genre` - equality, далі діапазон по `danceability`, далі `popularity` для сортування у спадному порядку.

**Після індексу:**

```
winningPlan.stage:                  FETCH
winningPlan.inputStage.stage:       SORT
winningPlan.inputStage.inputStage.stage:     IXSCAN
winningPlan.inputStage.inputStage.indexName: idx_genre_dance_pop
nReturned:                          354
totalDocsExamined:                  354         ← у 322 рази менше
totalKeysExamined:                  354
executionTimeMillis:                2           ← у 44 рази швидше
```

**Що змінилося:**
- `COLLSCAN` → `IXSCAN` - повне сканування зникло, Mongo читає тільки індекс.
- `totalDocsExamined`: **113 999 → 354** (рівно стільки, скільки реально повертаємо).
- `executionTimeMillis`: **88 мс → 2 мс**.
- `SORT` залишається в плані, бо `danceability` як range «розбиває» порядок ключів - це очікувано і прийнятно, бо сортуються вже 354 відібраних документи, а не всі.

**Як зрозуміти, що індекс використовується:**
- У `winningPlan` з'являється стадія `IXSCAN`.
- Поле `indexName: 'idx_genre_dance_pop'` явно вказує наш індекс.
- `totalKeysExamined > 0` (тут - 354).
- `totalDocsExamined` ≈ `nReturned` - індекс точно відбирає документи.

### Завдання 2. Індекс для фонової музики

```javascript
db.tracks.createIndex(
  { explicit: 1, "audio_features.speechiness": 1, "audio_features.instrumentalness": 1 },
  { name: "idx_work_music" }
);
```

Порядок підібрано за ESR: `explicit` - equality (boolean відсікає ~90% документів одним кроком), далі два range-поля.

**`explain()` після створення:**

```
winningPlan.stage:                FETCH
winningPlan.inputStage.stage:     IXSCAN
winningPlan.inputStage.indexName: idx_work_music
nReturned:                        16141
totalDocsExamined:                16141
totalKeysExamined:                16921
executionTimeMillis:              44
```

Індекс використовується: `IXSCAN` з нашим іменем, `totalKeysExamined: 16921` ≈ `totalDocsExamined: 16141` (різниця 780 ключів - це проміжні значення індексу, що не пройшли діапазонний фільтр). Без індексу Mongo прочитав би всі 113 999 документів.

### Завдання 3. Чи покривний запит?

Запит:
```javascript
db.tracks.find({ track_genre: "pop", popularity: { $gte: 70 } });
```
з індексом `{ track_genre: 1, "audio_features.danceability": 1, popularity: -1 }`.

**Відповідь: оригінальний запит - НЕ покривний.** Але якщо додати правильну проєкцію - стає покривним.

Покривний запит - це той, для якого Mongo читає всю необхідну інформацію **тільки з індексу**, без стадії `FETCH`. Для цього мають виконуватись три умови:

1. **Всі поля фільтра** є в індексі - (`track_genre` і `popularity` обидва присутні в індексі, серединне поле `audio_features.danceability` Mongo просто пропускає, скануючи його як повний діапазон у межах `track_genre: "pop"`).
2. **Всі поля проєкції** є в індексі - у запиті проєкції **немає**, тобто Mongo повертає **весь документ** (track_name, artists, audio_features тощо). Цих полів в індексі немає → Mongo змушений робити `FETCH` ✗.
3. **`_id` явно виключений** через `{ _id: 0 }`, бо `_id` повертається за замовчуванням, а в індексі його немає ✗.

**Перевірка експериментом** (у `part4_indexes.js` запит виконано з проєкцією `{ _id: 0, track_genre: 1, popularity: 1 }`):

```
winningPlan.stage:                PROJECTION_COVERED   ← ось воно
winningPlan.inputStage.stage:     IXSCAN
indexName:                        idx_genre_dance_pop
nReturned:                        317
totalDocsExamined:                0                    ← НУЛЬ! документи не читались
totalKeysExamined:                620
executionTimeMillis:              1
```

`totalDocsExamined: 0` і `stage: PROJECTION_COVERED` - це підтвердження покритого запиту: Mongo взагалі не звертався до документів, всі дані витяг з індексу.

**Висновок:** запит у тому вигляді, як написано (`db.tracks.find({...})` без проєкції) - **некритий**, бо потрібно повертати весь документ. Він стає покритим лише з явною проєкцією `{ _id: 0, track_genre: 1, popularity: 1 }`.
